Runtime Generation

Runtime generation lets you execute the same authored graph logic during play mode and turn the resulting ElementSet into live Unity objects. The standard entry point for that workflow is the ProceduralGraphExecutor component.

What happens at runtime

  • An execution context is created with OctoShaper's default runtime services.
  • A graph executor is created for the selected graph.
  • The graph runs with the current ProceduralConfiguration overrides.
  • The main output ElementSet is passed to the Unity instancing path to spawn scene objects.

Why generated execution matters

The default runtime bootstrap uses the strict generated executor path. That keeps runtime execution aligned with generated graph code rather than relying on editor-only conveniences.

If the generated registry is missing or stale, regenerate graph code before expecting runtime execution to succeed.

Choosing the right runtime surface

  • Use ProceduralGraphExecutor when you want live scene generation from a component.
  • Use a procedural prefab when you want an imported asset that regenerates from graph inputs.
  • Use synchronous execution for immediate deterministic results, or enable async execution at runtime when the graph's node implementations benefit from it.

Practical guidance

Keep the runtime graph boundary clean: main input in, main output out, and parameter overrides controlling the parts of the graph that need to react to gameplay. If you expect the runtime path to be robust, treat graph code generation as part of your normal authoring cycle rather than a final cleanup step.

When something goes wrong at runtime, the first things to check are generated graph code, the main output type, and whether the graph depends on editor-only assumptions that do not exist in play mode.
